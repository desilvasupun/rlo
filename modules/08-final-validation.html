Module 8: Final Validation and Evidence Package

Confirm that every layer of the stack works and gather a clean evidence package.

Estimated time: 10 minutes
Module progress
0 of 0 checkpoints completed

What you will complete

  • Validate the entire stack from OS to application.
  • Confirm the web, database, and automation layers still work together.
  • Organize the full evidence package captured in Modules 1 to 7.

Verification checklist

  • All required services are running.
  • The site loads, reports a successful database connection, and the Python health check returns HTTP 200.
  • Your evidence package is organized and includes the screenshots captured in Modules 1 to 7.

Steps

Run the full-stack validation sequence

Validate the environment from the bottom up. If a step fails, stop there and fix that layer first rather than skipping ahead. This makes troubleshooting faster and more systematic.

1. Confirm the operating system baseline

  1. Reboot the VM if needed and confirm Ubuntu reaches the normal login prompt.
  2. Log in with your non-root administrative account.
  3. Confirm the machine still has a valid IP address.
    Display the current IP address
    hostname -I

2. Confirm remote access and firewall state

  1. Check that the ssh service is running.
    Check the SSH service status
    sudo systemctl status ssh --no-pager
  2. Confirm your SSH hardening file still disables direct root login.
    Review the SSH hardening file
    sudo cat /etc/ssh/sshd_config.d/60-hardening.conf
  3. Run sudo ufw status verbose and verify the firewall is active.
    Check firewall status
    sudo ufw status verbose
  4. Make sure both the OpenSSH and Apache rules are still present.

3. Confirm the web and database services

  1. Check that Apache is running.
  2. Check that MariaDB is running.
    Check key service status in one pass
    systemctl --no-pager --type=service --state=running | egrep "ssh|apache2|mariadb"
  3. Query the database as lampp_user and confirm the seeded row is returned. After you run the command, MariaDB prompts for the password assigned to lampp_user.
    Test database access as the app user
    mariadb -u lampp_user -p lampp_demo -e "SELECT * FROM app_info;"

4. Confirm the application layer

  1. Open http://127.0.0.1:8080/ in a browser.
  2. Confirm the site loads without a PHP error or blank page.
  3. Confirm the page displays Database connection succeeded.
  4. Confirm the seeded application record is visible on the page.

5. Confirm the automation layer

  1. Activate the Python virtual environment again.
  2. Run the health-check script from Module 7.
    Run the Python health-check again
    cd /opt/lampp-demo-python && source .venv/bin/activate && python check_site.py
  3. Confirm it prints an HTTP 200 response and a short HTML preview.

Export a clean evidence package

Prepare the full evidence package while the system is still running and verified. Use the screenshots and other evidence you captured as you completed Modules 1 to 7, then organize them clearly for submission.

1. Compile the full evidence captured in Modules 1 to 7

  1. Gather the screenshots and other evidence you captured while completing Modules 1 to 7.
  2. Use the Evidence to capture list in each earlier module as your checklist for what must be included.
  3. Submit the full set of required evidence from each module.

2. Rename the files consistently

  1. Use filenames that begin with the module number so the evidence matches the build order.
  2. If a module has multiple screenshots, keep them all and add a second number so the sequence stays clear.
  3. Using this naming pattern, the Module 1 evidence could be:
  • 01-01-vm-summary.png
  • 01-02-install-complete.png
  • 01-03-login-success.png
  • 01-04-hostname-ip.png
  • 01-05-ping-ubuntu.png

3. Do a final quality check

  1. Compare your evidence folder against the evidence lists in Modules 1 to 7 and confirm that every required item is present.
  2. Open each screenshot and make sure the text is readable.
  3. Remove duplicate or blurry captures, but do not remove required evidence just because it is similar to another screenshot.
  4. Submit the organized evidence package as a zip file.
