Module 8: Final Validation and Evidence Package

Confirm that every layer of the stack works and gather a clean evidence package.

Estimated time: 10 minutes
Module progress
0 of 0 checkpoints completed

Your checklist progress is stored in your browser only on this device.

What you will complete

  • Validate the entire stack from OS to application.
  • Identify common issues systematically.
  • Finish with a reusable server build checklist.

Verification checklist

  • All required services are running.
  • The site loads and reports a successful database connection.
  • Your evidence package contains at least one artifact for each major module.

Steps

Run the full-stack validation sequence

Validate the environment from the bottom up. If a step fails, stop there and fix that layer first rather than skipping ahead. This makes troubleshooting faster and more systematic.

1. Confirm the operating system baseline

  1. Reboot the VM if needed and confirm Ubuntu reaches the normal login prompt.
  2. Log in with your non-root administrative account.
  3. Confirm the machine still has a valid IP address.

2. Confirm remote access and firewall state

  1. Check that the ssh service is running.
  2. Confirm your SSH hardening file still disables direct root login.
  3. Run sudo ufw status verbose and verify the firewall is active.
  4. Make sure both the OpenSSH and Apache rules are still present.
Check firewall status
sudo ufw status verbose

3. Confirm the web and database services

  1. Check that Apache is running.
  2. Check that MariaDB is running.
  3. Query the database as lampp_user and confirm the seeded row is returned.
Check key service status in one pass
systemctl --no-pager --type=service --state=running | egrep "ssh|apache2|mariadb"
Test database access as the app user
mariadb -u lampp_user -p lampp_demo -e "SELECT * FROM app_info;"

4. Confirm the application layer

  1. Open http://VM-IP/ in a browser.
  2. Confirm the site loads without a PHP error or blank page.
  3. Confirm the page displays Database connection succeeded.
  4. Confirm the seeded application record is visible on the page.

5. Confirm the automation layer

  1. Activate the Python virtual environment again.
  2. Run the health-check script from Module 7.
  3. Confirm it prints an HTTP 200 response and a short HTML preview.
Run the Python health-check again
cd /opt/lampp-demo-python && source .venv/bin/activate && python check_site.py

Export a clean evidence package

Organize the evidence while the system is still running and verified. A tidy evidence package is easier to grade, review, and reuse later.

1. Gather one clear artifact per major module

  1. Include the VM login or IP evidence from Module 1.
  2. Include the SSH or firewall evidence from Module 2.
  3. Include the Apache page or config test evidence from Module 3.
  4. Include the MariaDB query evidence from Module 4.
  5. Include the PHP execution or final site evidence from Modules 5 and 6.
  6. Include the Python health-check evidence from Module 7.

2. Rename the files consistently

  1. Use filenames that begin with the module number so the evidence matches the build order.
  2. A clean naming pattern is:
  • 01-vm-ip.png
  • 02-firewall-status.png
  • 03-apache-page.png
  • 04-mariadb-query.png
  • 05-php-site.png
  • 07-python-check.png

3. Do a final quality check

  1. Open each screenshot and make sure the text is readable.
  2. Remove duplicate or blurry captures.
  3. Keep a short notes file with any passwords, IP addresses, or explanations that your submission requires.

Reflect on what makes this reusable

Finish by identifying why this build process can be repeated by another learner or reused for a future class project.

1. Summarize the reusable structure

  1. Note that the build follows a logical order: operating system, hardening, web server, database, PHP application, then automation.
  2. Note that each module includes a verification checkpoint so the learner can detect errors early.
  3. Note that the commands are copy-friendly and the configuration is simple enough to reproduce in a fresh VM.

2. Distinguish baseline deployment from advanced production work

  1. Record that this RLO covers a production-oriented baseline, not every enterprise feature.
  2. Examples of out-of-scope advanced topics include TLS certificate management, load balancing, high availability, centralized monitoring, and CI/CD pipelines.
  3. This keeps the tutorial focused and repeatable for the intended audience.

Troubleshooting

IssueWhat to check
Only Apache works but PHP or DB failsRetest each layer separately: shell package status, Apache logs, MariaDB login, then application code.
The stack worked earlier but not after a rebootCheck that services were enabled to start automatically and that the VM networking mode still provides an IP address.
You need to repeat the build laterReuse the module order and command blocks, but update package versions and the Ubuntu LTS release if needed.
